【方案规划】项目目录深度治理与统一部署规范

主创：高级全栈架构师 | 阶段体系：Stage 1 - Align & Architect

📌 1. 核心痛点与环境关联诊断

  • 根目录重度污染：堆积大量散乱的 .sh 脚本、.md 说明文档、归档包（*.tar, *.zip）与数据 SQL，导致文件索骥成本极高。
  • 防范网破洞：先前的 .dockerignore 及源码同步脚本使用了词汇泛匹配 data/logs，误伤屏蔽了工程必须层 web/src/data，这导致构建流程强行阻滞。
  • 单向分化与自动化断层：开源侧 github-sync 尚未制定分离壁垒同步原则；项目缺乏集成 “打包-文件校验-传输-构建热更” 一体化防挂主控引擎。

📐 2. 目录重整体系与逻辑划分映射

实行“绝对隔离，高内聚低耦合”分类沉淀：

        graph TD
            Root["/ (根目录基座)"] --> Docs["/docs (管控所有架构方案、Plan及其它子级说明书)"]
            Root --> Scripts["/scripts (集中存放隔离后的所有 Shell 逻辑)"]
            Scripts --> SDeploy["/scripts/deploy (部署引擎、环境网络预装联通脚本)"]
            Scripts --> SUtils["/scripts/utils (修复密码、依赖清理、其他日常辅助)"]
            Root --> Arch["/.archives (隐蔽的打包容器、历史日志归属地，常驻阻断名单防上传)"]
            Root --> GithubSync["/github-sync (双生隔离库：剔除敏感隐私文件的映射极净仓库)"]
            Root --> Configs["保留零碎原位配置 (package.json, Dockerfile)"]

🛡️ 3. 部署坑点规避机制预设与影响面(Impact)评估

  • 硬链接级隔离规则：文件阻断清单强制由原本的自由泛匹配更改为绝对定解路径（例如只对 /data 或 /core/data 截留），杜绝次级重要结构由于同名被无情绞杀！
  • 纯净依赖抗灾筑基：后续生成的一键起服及新 Dockerfile 将从底层绑定好所有的加速基石（包括注入阿里系 npmmirror & alpine apk mirrors）。
  • 运维影响面副作用：部分开发者习惯于通过在主目录直接运行 ./quick-start.sh 的起服旧路将被截断。对于此副作用，后续我们将对外提供统一聚合启动脚本（包裹器），同时必须修正定时 Crontab 里指向老的路由信息。
