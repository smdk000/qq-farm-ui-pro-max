3. 高风险点与优先级

以下 16 个点建议作为首轮测试的强制覆盖项。只要其中任何一项失败，都应该阻断上线或至少阻断对外发布。

P0 必测 P1 次级关键 P2 增强覆盖
P0-01 空用户名登录分支

/api/login 中存在管理员特殊分支，需要确认空用户名不会绕过正常认证逻辑。

P0-02 普通用户越权访问后台页

前端菜单隐藏不代表安全，必须直接访问路由和直接请求 API 进行双重验证。

P0-03 普通用户设置间隔越界

普通用户低于 15 秒/60 秒的轮询配置应被后端拒绝，而不是仅前端警告。

P0-04 扫码登录状态机

二维码的 Wait、Check、OK、Used、Error 分支要全部跑通。

P0-05 重复账号添加

同一 uin + platform 新增时，后端会尝试转成更新；要验证是否会错误覆盖备注或生成重复账号。

P0-06 体验卡账号数上限

体验卡用户应被 maxAccounts 限制；新增账号时必须验证后端是否真正阻断。

P0-07 修改密码规则不一致

管理员和普通用户密码规则不同，必须分别验证前端和后端是否一致。

P0-08 刷新 token 与页面恢复

401 后前端会尝试刷新；要确认成功时自动恢复，失败时清理状态并跳回登录。

P1-09 主题配置“静默成功”

普通用户保存全局主题和下线提醒可能返回成功但不落库，需要验证视觉状态与真实配置是否一致。

P1-10 好友列表缓存回退

实时获取好友失败时，是否能正确回退缓存且给出准确提示，不出现空白页或误导状态。

P1-11 偷菜过滤优先级冲突

好友黑白名单、植物黑白名单、跳过白萝卜、蹲守偷菜叠加时，最容易出现业务逻辑错误。

P1-12 工作流保存与回显

流程编排既有前端拖拽状态又有后端配置快照，必须确认刷新、切换账号、重新登录后不丢节点。

P1-13 扫码后 wsError 防抖

账号保存后的 8~15 秒窗口内若再出现 wsError，是否会错误弹出账号窗。

P1-14 试用卡双重限流

既有公开 API 的请求频率限制，又有 IP 冷却和每日总量，必须分别验证。

P1-15 系统日志查询边界

筛选、分页、关键词、查看详情时要确认不会因字段为空或 JSON 异常导致界面崩溃。

P2-16 多账号长时间运行

这是后续稳定性关键指标，建议在手工通过后再补自动化和长期运行监控。

Phase 0 环境与基线 0.5 天

准备测试环境、账号、卡密、体验卡、第三方依赖、日志留痕方式，确保后续问题可复现。

  • 确认管理员基线账号、普通用户、过期用户、封禁用户、体验卡用户。
  • 准备可用卡密、已使用卡密、禁用卡密、无效卡密。
  • 确定第三方扫码接口的可测性与不可控依赖点。
  • 建立缺陷记录模板与测试日志模板。
Phase 1 认证与权限 1.0 天

先验证系统“能否安全地进来”和“是否只能看到自己该看的东西”。

  • 登录、注册、免责声明、记住用户名、退出、token 刷新。
  • 普通用户和管理员的菜单差异与直接访问差异。
  • 卡密、用户、公告、系统日志等后台接口的越权验证。
  • 登录锁定、体验卡公开接口限流。
Phase 2 账号与扫码登录 1.0 天

这是项目主链路，必须覆盖新增、更新、自动启动、重复账号、扫码状态机和并发误操作。

  • 手动填码与四种扫码平台分支。
  • 运行中账号更新后的自动重启逻辑。
  • 风险规避模式、一键黑名单、删除与停止。
  • 体验卡用户账号数量上限。
Phase 3 设置中心全量验证 1.5 天

逐项验证设置项的有效、无效、边界、组合错误和保存后的实际生效，避免出现“只是回显成功”。

  • 账号模式、种植策略、偏好种子、间隔、静默时段。
  • 自动化开关、施肥策略、蹲守偷菜、好友/植物过滤。
  • 下线提醒、主题、体验卡配置、第三方 API、集群配置、时间参数。
  • 切换账号、刷新页面、重新登录后的回显一致性。
Phase 4 好友 / 偷菜 / 工作流 1.0 天

这一阶段主要查逻辑错误和状态同步问题，特别是过滤优先级和工作流节点配置。

  • 好友列表实时加载与缓存回退。
  • 好友黑名单、植物黑白名单、好友黑白名单、跳过白萝卜。
  • 工作流节点拖拽、删除、参数编辑、保存、回显。
  • 账号未登录或切换账号时的行为。
Phase 5 后台管理与可观测性 1.0 天

验证管理员能力链路，包括用户、卡密、公告、系统日志、通知、排行榜等。

  • 用户封禁、解封、改到期时间、体验卡续费。
  • 卡密生成、编辑、删除、批量删除、使用状态检查。
  • 公告管理、通知列表、系统日志分页与详情。
  • 排行榜、账号日志、仪表盘实时日志过滤。
Phase 6 鲁棒性 / 安全 / 性能 1.0 天

通过故障注入和异常输入找脆弱点，防止系统上线后在边缘条件下崩溃。

  • 断网、超时、401、403、500、socket 断连重连。
  • localStorage 污染、多标签页冲突、快速连点、重复提交。
  • 登录暴力尝试、试用卡刷接口、伪造管理员请求。
  • 多账号并发、长时间运行、日志压力。
Phase 7 自动化与最终验收 1.0 天

将最关键用例固化为自动化，减少回归成本，并产出最终缺陷闭环报告。

  • 定义 API 集成测试、E2E 测试和冒烟回归集合。
  • 固化 P0 用例，建立版本回归基线。
  • 提交最终测试报告、缺陷列表、回归建议。

6. 测试数据与环境准备

为了避免“测不出来”或“测完无法复现”，建议一开始就准备以下测试资产。

6.1 测试账号

  • admin-01：默认管理员或已修改密码的管理员。
  • user-01：正常付费用户。
  • user-02：体验卡用户，接近账号上限。
  • user-03：已过期用户。
  • user-04：已封禁用户。

6.2 卡密样本

  • 可用天卡、周卡、月卡、永久卡、体验卡各至少 1 组。
  • 已使用卡密 1 组，禁用卡密 1 组，随机无效卡密 1 组。
  • 体验卡每日额度打满和 IP 冷却中的场景样本。

6.3 账号与好友场景

  • 至少准备 2 个可正常启动的农场账号。
  • 准备一组好友列表缓存数据，便于验证回退逻辑。
  • 准备包含白萝卜和非白萝卜作物的好友地块样本。

6.4 故障注入场景

  • 网络断开、第三方扫码接口超时、后端返回 401/403/500。
  • 清除或篡改 localStorage。
  • 高频连续点击与重复提交。

7. 交付物与验收标准

如果这项工作交给我来执行，我会把最终交付物控制为“能指导修复、能指导回归、能指导上线”的三类结果，而不是只给一句“测过了”。

交付物 01 测试执行台账

记录每个任务 ID 的执行状态、测试日期、执行人、环境、结果和备注。

交付物 02 缺陷报告

至少包含严重级别、复现步骤、预期结果、实际结果、截图/日志、影响范围。

交付物 03 复测与回归结论

修复后复测结果、残余风险、建议阻断项和建议放行项。

交付物 04 自动化候选清单

明确哪些用例应该优先固化为自动化，降低后续版本回归成本。

7.1 验收标准

P0 标准：P0 用例全部执行完毕，且不存在未接受的 P0 缺陷。
P1 标准：P1 缺陷要么修复，要么有明确风险说明和上线决策。
P2 标准：P2 可以延后，但必须进入回归或自动化 backlog。
最终结论：给出“可上线 / 有条件上线 / 不建议上线”三选一结论。
