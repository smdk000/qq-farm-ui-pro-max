🚀 系统架构与稳定性深化改进计划

📌 1. 背景与核心问题综述 (The Context & Issues)

在近期进行了大量底层防封、UI多主题重构及多账号模式等高强度跨域优化后，代码库已趋近于 4.1.1 稳态版本。但在极端测试与高维压力推演下，我们发现 3 个可能会在未来或者高频多账号并发集群中爆发的严重隐患：

问题源 等级 隐患描述 触发条件
三阶段偷菜机制 高危 (风控) checkFriendsThreePhase 及手动操作会在网格偷取后立即无条件调用 sellAllFruits()。 好友列表极长且果实成熟频次高时，极易触发腾讯接口短时 限流与风控黑洞。
前端全景面板 中危 (性能) Web 端在接收 WebSocket 发车队列推送时缺乏截流机制。 20+ 开账号“全部启动”瞬间，任务疯狂推入可能打满 Chrome 内存与主线程，导致 假死/白屏。
流程编排 (Workflow) 中危 (死锁) 移除 autoPlantEmptyLands 内置化肥调用后，系统完全依赖流程图中的节点。 用户小白误开启流程编排却 漏配 stage_fertilize 节点，植物将干枯无法成长。

🧩 3. 详细实施路径 (Execution Steps)

阶段 1: 剥离售卖逻辑，实施旁路守护 (Anti-Ban Isolation)

  • 目标文件: core/src/services/friend.js, core/src/services/farm.js (可选)
  • 动作:
    1. 移除 visitFriend, checkFriendsThreePhase 和 doFriendOperation 成功偷菜后强制跟进的 await sellAllFruits()。
    2. 在内存状态管理 runtime-state.js 或 store.js 中新增标记槽位位 accountsNeedsSell: Set<gid>，偷菜成功仅推入 gid 标记。
    3. 在 scheduler.js 中挂载一个每 15 分钟（加随机偏移）触发一次的 Warehouse Sell Daemon 守护进程，消费 accountsNeedsSell 并执行缓释贩卖，大幅解耦网络请求。

阶段 2: 前端高频重绘防抖降频 (Vue Reactivity Throttle)

  • 目标文件: web/src/stores/app.ts 或 web/src/views/Dashboard.vue
  • 动作:
    1. 引入 VueUse 的 useThrottleFn 或者手写截流定时器。
    2. 在接收 WebSocket 的 on('task_queue_update', data) 事件回调时，不再直接 state.tasks.value = data，而是压入过渡缓存对象，利用 Throttle 控制至少 500ms 才向 Vue 响应式数据推送一次更新。
    3. 结合已有的 DOM 虚拟化，彻底杜绝瞬间万件 DOM 更新造成的浏览器卡崩。

阶段 3: 流程编排完整性约束断言 (Workflow DAG Assertion)

  • 目标文件: web/src/views/Settings.vue, core/src/services/config-validator.js
  • 动作:
    1. 前端强提醒: 在用户勾选 启用流程编排 时，解析其 workflowConfig.nodes，如果检测不到 type === 'stage_fertilize'，弹窗提示用户：“您的编排图中缺少施肥节点。由于已禁用传统施肥，可能导致植物生长极度缓慢甚至缺肥停滞，是否确认保存？”
    2. 后端防漏校验 (可选增强): 在 /api/settings/save 通过 schema validator 添加对于 workflowConfig 完整性的 Warning 级别透传。

💣 4. 影响面评估 (Impact Analysis)

  • 数据一致性与持久化: 新建的售卖守护任务必须正确依托现存的 tokenBucket 频率限制网关，不能打破目前多号并联使用的接口安全速率。
  • 用户体感变化: 用户将发现在 Dashboard 点击“偷菜”后，果实不会像以前一样立刻变现为金币而是存在着“延迟结算”感。需要在控制台 Logs 打印清晰提示：“已寄存至自动售卖队列，稍后将由旁路守护统一结算。”
  • 视图刷新率折损: 增加防抖会牺牲视觉层面的“绝对实时（0ms级跳跃）”，但是换来了 UI 主进程如丝般顺滑的操作手感，收益远大于代价。
