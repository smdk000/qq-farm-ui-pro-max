系统演进规划书 (Phase 5: Stats, Perf & Distributed)

文档目标： 针对 Phase 4 遗留的三个核心建议（数据持久化打通、Echarts 前端按需加载优化、中继分布式架构演进）制定详尽的落地方案、架构设计与影响面分析。

一、 真实战报打通：MySQL stats_daily 持久化与 CRON 定时聚合

目前 /api/stats/trend 的数据来自模拟抖动。为了实现真实数据监控，我们需要利用 MySQL 建立日结表，并使用 CRON 或后台定时任务执行清算。

1. 数据库结构设计 (Schema)

CREATE TABLE IF NOT EXISTS stats_daily (
    id INT AUTO_INCREMENT PRIMARY KEY,
    record_date DATE NOT NULL,
    total_exp INT DEFAULT 0,
    total_gold INT DEFAULT 0,
    total_steal INT DEFAULT 0,
    total_help INT DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY `uk_date` (`record_date`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

2. 数据流与 CRON 处理机制

  • 内存态累加：复用现有 stats.js 中的 session 或 operations 对象，将生命周期内的涨幅存入内存或 Redis。
  • CRON 清算 (Daily Job)：每日 23:59:50 触发定时任务，将当日所有 Worker 的内存统计增量 UPSERT (INSERT ... ON DUPLICATE KEY UPDATE) 写入 stats_daily。
  • 前端读取：/api/stats/trend 从该表执行 SELECT ... ORDER BY record_date DESC LIMIT 7。

二、 前端按需加载优化：Echarts DefineAsyncComponent

由于 Echarts 图表库体积巨大，直接引入会造成首页 (Dashboard) 加载时网络阻塞（即便用户没进入作战大屏页面）。我们需要在 Vue 路由或组件层进行拆分。

1. 优化策略实施代码

在 router/menu.ts 或外层组件中，必须确保 component: () => import('@/views/AnalyticsEcharts.vue') 是异步的（目前已经是）。但是如果在内部还需要更细粒度的包拦截，可以使用：

import { defineAsyncComponent } from 'vue'

const VChart = defineAsyncComponent(() => import('vue-echarts'))
// 在 AnalyticsEcharts.vue 中避免顶层直接执行 echarts.use，而是移入 onMounted 或 Async 包内

2. 性能提升表现 (Impact)

  • Vendor Chunk 分离：通过 Vite 的 manualChunks 配置，将 echarts 单独打包为一个异步 JS 文件（如 vendor-echarts.js）。
  • 首屏提升：将首页 Dashboard.vue 的首屏加载体积缩减至少 1.5MB (Unzipped)，降低白屏 TTI 时间。

三、 架构跃迁：多级分布式协同挂机 (Replay Relay)

突破单机单线程限制，设计“一个主控端调度分配，多个边缘节点打工”的全新分布式挂机集群架构。

1. 架构流转图 (Mermaid)

graph TD A[用户 UI 面板] -->|增删改查账号 & 策略| M(Master 主控中心) M -->|MySQL / Redis 持久化| DB[(中心数据库)] M -->|RPC/WS 任务调度| MQ((任务队列分配器)) MQ -->|下发: 账号A_Token/配置| W1[Worker 轻量节点 1] MQ -->|下发: 账号B_Token/配置| W2[Worker 轻量节点 2] MQ -->|下发: 账号C_Token/配置| W3[Worker 轻量节点 N] W1 -.->|上报运行日志/Stats| M W2 -.->|上报运行日志/Stats| M W3 -.->|获取 1002003 触发熔断指令| M W1 ---|腾讯接口请求| API(腾讯 QQ 农场 API) W2 --- API W3 --- API

2. 核心模块与实现路径

  • Master 中心管控层：运行全量 Admin UI 服务、数据库连接以及中继调度中心。提供给下级节点的通讯 API。
  • Worker 边缘工作层：剥离了 UI 服务的纯粹 Node.js 脚本。启动时通过 Token 链接 Master，拉取属于自己执行的账号列表（appConfig）。
  • 分布式锁与健康侦测（Heartbeat）：各个 Worker 每隔 30 秒向主控汇报活跃状态。如果节点掉线，主控把该节点下的账号释放，重新分配给其他空闲 Worker（故障转移 Failover）。
  • 全局熔断器下发：如果在 Worker 1 触发大规模 1002003，它立刻通知主控，主控通过广播向所有 Worker 下达短路挂起指令。

3. 影响面分析 (Impact Analysis)

  • 研发复杂度： 极高。需要重构 daemon 进程，引入 Redis 流或 RabbitMQ 概念（可用简易 WS 替代）。
  • 风控阻隔： 将不同账号摊派在不同的 VPS (不同 IP 外网出口) 上运行，能彻底规避同 IP 出口高频调用引起的集团连坐封禁。
  • 资源利用： Master 完全可放置在轻薄云上，Worker 可运行在各种局域网闲置 PC、树莓派等设备。

四、 执行建议与优先级 (Priority)

  1. [ P0 - Immediate ] Echarts 按需加载优化：改动最小，见效最快，仅需微调 Vite 构建和组件引入规则。
  2. [ P1 - Short Term ] 真实战报打通：为现有的 Echarts 大屏填充灵魂，需新增一张表，并写一段定时收集逻辑。可以在接下来开发中顺手完成。
  3. [ P2 - Long Term ] 分布式协同挂机：属于大版本跨越（如 v5.0 甚至 v6.0 的核心基石）。需要单独起一个新项目分支（如 next-dist-arch）进行概念验证 (POC)。
