🚀 农场引擎演进 Phase 6: 分布式 Master-Worker 架构设计蓝图

核心愿景：突破 NodeJS 单核运行瓶颈，终结随着账号膨胀导致的 GC 卡顿与并发延迟。将系统架构升级为“控制面 (Control Plane) 与 数据/执行面 (Data Plane/Workers) 分离”的跨服务器集群运作体系。

1. 业务痛点分析溯源

在 Phase 4 和 Phase 5，我们连续打通了全局断路器、Echarts 可视化大屏、WebSocket 的 Room Pagination 分页以及 MySQL 实时资产聚合。所有的底层应用级安全隔离带与数据基座已修筑完毕。

目前的瓶颈：

  • 集中式瓶颈：所有的爬虫（巡查、签到、好友请求）都堆积在唯一的主进程里。账号量达到千级别时，事件循环池将极度拥堵。
  • IP 风控隔离：单 IP 承载过多账号必然触发 1002003 极严厉风控。
  • 扩容僵硬：无法动态水平扩展机器。

责任清晰化：

  • Master：不跑任何游戏业务。专职收发用户指令、读取数据库并分配账密、渲染面板（Analytics）、持久化每日收益聚合。
  • Worker：极度轻量化，没有数据库概念，接到 Master 分配的 Account 结构体后，立刻执行轮询。向 Master 的 API 发送心条包反馈日志。

3. 核心分阶段重构点 (Epic 拆解)

第一梯队：数据与调度分层 内部改造

方向 动作描述 说明
Worker 通信协议 建立 Master <=> Worker 的双向通信。推荐初期使用长轮询 (Long Polling) 结合 Heartbeat，或在 Worker 与 Master 间建立轻量级 WS 隧道。 方便 Worker 将 log:new 和 status:update 回推给主节点存入内存池。
数据剥离结构 剥离 farmer-agent.js / friend-scanner.js，使其作为纯粹的 Worker 类。 通过环境变量 ROLE=master 或 ROLE=worker 决定启动的服务类。

第二梯队：无状态执行节点设计 基础建设

  • 所有需要通过网络请求（Axios）向 Aineishe或 QQ 发起的函数，必须放在 Worker 内发生。
  • 实现分布式锁或简易的一致性 Hash 算法（例如按 QQ 号长度取模运算，分发给挂载上来的 Worker_N）。如果 Worker 突然掉线（心跳丢失超过 1 分钟），Master 自动将其名下的账号转移给存活的其它 Worker。

第三梯队：容器化与 CI/CD 改造 部署基建

  • 扩写目前的 docker-compose.yml 实现模拟测试。新增 farm-worker 服务进行容器复用缩放 (e.g. docker-compose up --scale farm-worker=3)。
  • 由于每个 Worker 最终大概率会被分配到不同的物理机器来使用不同的家用宽带 IP（抗风控），所以需要配置外部 Worker 连接 Master 的安全握手鉴权（类似于目前系统中的 token 机制，但针对集群）。

4. 潜在风险及应对方案 (Impact & Mitigation)

  • 数据延迟 (Data Staleness)：前端可能无法像目前一样极致的看到 ms 级的 100% 实时反馈。
    应对：接受合理的延迟（如 2-5秒同步），在页面用骨架屏掩盖视觉延迟。
  • 日志收集海量洪峰：Worker 将所有明细发回 Master。
    应对：我们已经在 Phase 5b 完成了 Room Pagination（分布式 WS 频道），这为集群洪峰起到了决定性的降压。对于后台内部，可以在 Worker 端实行 10秒 批量提交机制，压缩网络交互。
