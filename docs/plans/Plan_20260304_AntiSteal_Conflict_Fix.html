QQ农场智能助手 - 功能架构优化计划 (Plan)

构建时间：2026-03-04

核心目标：解决「极限防偷抢收」与「全局防封休眠锁」、「小号多级错峰」之间的优先级干涉冲突，提升长期高等级防封的韧性。

1. 业务流程与架构冲突现状分析

高危 致命冲突点：当前防偷探测器 farmScheduler 是无视网络层状态的。如果账号在 1 分钟前刚好触发了腾讯 1002003 风控报错并被置入了 suspendUntil 长休眠，1 分钟后如果有菜成熟，系统依然会被唤起并发起化肥请求，这会极大概率引发黑名单的顺延与连坐。

2. 决议：状态与调度优先级重构

针对这一风险，我计划设计一套严密的「调度隔离屏障」，确保安全高于收益。

2.1 防封锁最高优先级保护机制 (Safety First Override)

  • 机制：修改运行引擎 (farm.js) 中的 checkAndHarvest 或者包含在防偷定时器的判定里。
  • 实施：拦截任何发向服务器的请求。在准备给待抢收农作物施放「普通化肥」之前，检查 account.state.suspendUntil 是否处于生效期。若在生效期内，宁愿被偷也绝不可打破休眠去下发包。

2.2 小号模式排异降级处理 (Alt Mode Exclusion)

  • 机制：既然小号本身是为了“提供碎片化收益”并降低在线峰值而定，防偷这种极端内卷竞争在它身上没有必要。
  • 实施：如果一个账号开启了 ALT 小号模式，则在 farm.js 里自动关闭对其「极限防偷功能」的监听循环，强制走最保守的时序路线。让它遵循原设定的 180s~300s 开外延迟再去收菜。

3. 对现网架构的影响面 (Impact Analysis)

  • 网络底座通信：无影响，利用已有的状态机节点变量（opUser.state.suspendUntil）。
  • UI 显示：无影响，但可能会发现小号比以前明显被偷得更多了，因为这符合设计本意（舍去防偷，保大号安全错峰环境）。
