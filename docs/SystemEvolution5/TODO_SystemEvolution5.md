## 1. 深度安全清算建议
- [ ] 当前的 CRON 任务会强制累加 `total_exp`。若遭遇极小概率的宿主机时间回调（NTP Time Shift）或程序发生崩溃导致反复重启，需要在写库逻辑内追加更深的 `Dirty Write` 标志位判定。

## 2. 交互体验再升级
- [ ] 随着分布式的建立或者账号规模上到百级甚至千级维度，目前全量拉取 `accounts` 传向前端容易打爆 WS（WebSocket）通道，产生类似“Array size limit”的越界。建议在日后重构前端 Pinia 和后端的 Status 发送通道时，引入分页订阅通道（Room Pagination）。

## 3. 第六阶段：分布式 Master-Worker 架构！
我们已经在《系统演进规划书 Phase 5》立项探讨了这部分，随着底层单机架构的加固完毕（错误捕获、应用限流、定时写库、大屏查看、微信降频等全部完成），下一级宏伟目标即是构建集群通信：
- **Master 分发层：** 取缔目前由单机包揽爬虫的职责，退居二线管理账号配置清单与监控大屏（Admin Panel）。
- **Worker 执行层：** 去中心化的微型 Docker 容器。只挂载 axios 与签到逻辑，向上级索要任务并回推日志。
