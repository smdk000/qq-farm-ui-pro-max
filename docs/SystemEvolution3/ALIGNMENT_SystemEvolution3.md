# ALIGNMENT: 农场系统演进 Phase 3 (深度解耦与可视化审计)

## 📌 背景与目标
在 Phase 2 我们完成了底层 ACID 防御和 QoS 队列，同时开启了 `system_logs` 的 MySQL 同步写入。
现在随着业务拓展，遗留的两座大山需在本期攻克：
1. **巨物文件之痛**：`friend.js` 逼近 2000 行，承担了扫描、决策、偷菜、帮忙、防风控等多重职责，违背了单一职责原则(SRP)，修改任何一处都会引发蝴蝶效应。
2. **审计瞎子问题**：目前虽然存下了全量日志，但只能通过后端连 Navicat 查询，非研发用户无法在遇到“不运行”、“封禁”时自行追溯这部分冷数据日志。

**本期任务目标：**
在维持现有业务运转的前提下，重构拆分好友模块，并打通全栈，为前端赋予“系统审计台”的能力。

## ⚠️ 约束与红线
- **拆分红线**：重构 `friend.js` 时必须保证其暴露在外部的接口（`startFriendCheckLoop`, `stopFriendCheckLoop`等）行为完全一致，且不可破坏新引入的 PlatformStrategy。
- **性能红线**：全量日志查询接口 `/api/logs` 必须提供分页（Page）和索流（Limit），绝对不可一次性 `SELECT * FROM system_logs` 卡死进程。
- **UI交互约束**：管理台使用现有的 `Ant Design Vue` 和暗黑 Tailwind 风格保持统一，采用弹窗或独立页面进行呈现。
