# ALIGNMENT: 农场系统演进 Phase 2 (可靠性与解耦重构)

## 📌 背景与目标
经过上一轮（Phase 1-4）的改造，系统的并发承压与反风控能力大幅提升，但也暴露出以下四个隐患亟待解决：
1. **数据写入锁**：单纯的 MySQL 查询未启用事务，重并发下依旧可能发生状态死锁。
2. **生硬丢弃策略**：普通的 5s 排队超时直接丢弃会导致 UI 出现空白重试黑洞。
3. **平台代码耦合**：业务层大量通过 `if (platform.startsWith('wx'))` 控制流程，代码腐化加剧。
4. **长效日志失忆**：前端 LRU 限制了 300 条后，历史故障无法在面板追溯。

**本次任务目标：**
将上述 4 点进行工程化重构填补，保证高可用性的同时恢复业务代码的优雅，并补全操作追溯能力。

## ⚠️ 约束与红线
- **拒绝过度设计**：不全面引入笨重的 Prisma/TypeORM 对象关系映射，而是封装精简的 MySQL Transaction Helper。
- **无感迁移**：日志解耦不应影响当前实况日志的渲染速度。
- **向下兼容**：策略模式替换过程中不能破坏原有 QQ 扫码端逻辑。
