# 统一认知文档 (ALIGNMENT) - 农场系统演进 Phase 4

## 业务背景与问题痛点
经过前三个 Phase 的技术债清还，我们在底层加固了令牌桶队列、消除了 MySQL 并发锁冲突、分离了微信防封平台策略，并完成了上帝类（`friend.js`）的物理隔离与可视化审计大屏。

然而，目前系统仍存在几项深水区挑战，迫切需要进入 Phase 4 彻底打通：
1. **被动挨打的风控**: 虽然有了安全日志 `system_logs`，但应用本身遇到大面积封禁报错时（如连续遇到 1002003 错误），无法**主动**停止运行。若不介入主动熔断机制，容易扩大全环境危险封号面。
2. **数据孤岛与表现力不足**: `stats` 追踪模块统计了当日执行的成功次数和经验增长情况，但只留在终端或零散面板上，老板或用户无法看到极具冲击力和直观的时间轴增长曲线（如经验与金币的每日长势）。
3. **Admin路由过度拥挤**: 随着历史的迭代，`admin.js` 中保留了大量针对 Farm Tools 的“本地计算” API（例如 `/api/calculator` 和 `/api/lands_for_level`）。这与主要的 Auth 和基础业务耦合在一起，不利于内存和进程资源的独立调度。

## 对齐目标 (Objectives)
- [x] **实现应用级断路器 (Circuit Breaker)**: 当识别安全事件阈值越线时，拦截并停止网络排队引擎 `network.js` 与 `friend.js` 的循环。
- [x] **引入数据大屏 (Vue + Echarts)**: 在概览页或状态页集成交互式的趋势图表（折线/柱状图），提供直观感官。
- [x] **拆分 Express 路由**: 将 `admin.js` 内涉及农场计算逻辑和衍生工具集的路由剥离至 `routes/farm-tools-routing.js`，实现微服务化的模块化挂载。

## 不做的事 (Out of Scope)
- 暂时不引入外部监控如 Prometheus / Grafana（保持零外部依赖，使用自身面板展示）。
- 不触碰底层登录扫码相关的授权凭证（Phase 2/3 已验证稳定性）。
