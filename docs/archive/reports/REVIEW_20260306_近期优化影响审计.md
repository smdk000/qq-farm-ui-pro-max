# 近期功能优化审查与影响评估报告

> **审查时间**: 2026-03-06
> **审查对象**: v4.1.0 - v4.3.0 系列重大架构变动与重构

## 1. 已完成的重大优化项盘点

1. **底层数据架构**: JSON 文件锁替换为 MySQL 并发连接池 (Phase 1)。新增了 `stats_daily` 的持久化每日清算 (Phase 5)。
2. **长连接队列防爆**: `network.js` 添加 5000ms 排队丢包防御；WebSocket 加设单页 `accountIds` 切片订阅保护 (Phase 2 & Phase 5)；LRU 限制前端状态树渲染规模。
3. **分布式算力解耦**: 彻底拆解单引擎应用，落地基于 `socket.io-client` 互相串联的 Master-Worker 集群模式并生成配套方案 (Phase 6)。
4. **统一平台逻辑**: 将早期的独立终端硬编码全部柔化，以 QQ `SyncAll` 和 WeChat `GetAll` 构建自动降级容灾。

## 2. 优化带来的潜在隐患与已知影响评估

虽然系统在防封与性能架构上取得了极大进展，但也引发并伴随几项系统性风险或已证实的影响：

### 2.1 数据库结构僵化引发的版本断层
**表现**: 本阶段强制切割了以原生文件保存的所有帐号资料并打入了 MySQL，如果有些旧版用户（尚未升级）从云端或本地更新此次集群代码，但其没有运行 `quick-start.sh` 或没有数据库根环境，服务将瞬间崩溃（Node.js `ECONNREFUSED` 或者抛出缺少 `stats_daily` 字段的 SQL Error）。
**建议**: 后续需要在启动流程 (`client.js`) 中增加更强的环境阻断探测，如果未正确加载数据库结构或旧依赖，直接打印出帮助指引阻止启动，而非让大量抛错淹没终端。

### 2.2 Master/Worker 的账号动态覆盖问题 (Hot Replace 缺陷)
**表现**: 目前 `master-dispatcher.js` 对 Worker 节点的发配方式为：直接 `emit('master:assign:accounts', accounts)`，Worker 收到后强制根据指令 `Set` 重刷所有的线程。
**影响**: 如果有几十个处于不同阶段成熟度的节点，如果在 Admin 页面随便增加或修改了一次小号的密码触发了后端的热分配操作，可能会将 Worker 端原先保存的临时排队/防抖 Timeout 全部无情强行切断，造成不必要的重连。
**建议**: 对于下一阶段，应开发基于 Account ID 的**差分重载（Diff Loading）**，Master 只对修改过的单点 `id` 发送热更新通知。

### 2.3 PM2 / Screen 等旧运维工具与新分布式角色的交叉混淆
**表现**: 目前的 `farm-bot.sh` 和 PM2 脚本依然以启动单机版 `.env` 的 `node client.js` 作为核心逻辑兜底。
**影响**: 从 v4.3.0 引入分布式 Docker 后，新手很容易在使用旧的 PM2 管理时错位叠加了多个 Master 节点（争抢同一个端口），导致环境崩坏。
**建议**: 在未来的运维迭代中，废除裸机脚本，实现全栈全渠道强制走 `docker-compose` 起服，简化开发精力发散。

### 2.4 Echarts 异步加载闪转带来的首次交互延迟
**表现**: 虽然 `defineAsyncComponent` 将巨大 JS 拆分，使得侧边栏切换瞬间不卡白屏，但点击「作战大屏」的头一两秒钟可能因为正在获取 JS Chunks 会带来轻微的“点击无反应”错觉。
**建议**: 可以增加一个统一的全局透明加载遮罩`<Suspense>`（带有简单的 `Loader/Spinners` 动画），在前端消除由于请求带来的“停顿迷茫期”。

## 3. 总体结论与下一步行动方向

系统目前正处于从**“个人玩物脚本”**正式蜕变为**“工业级千规模分布式的商业级爬虫平台”**的转折点。本轮引入的 `RateLimiter` 取模和 Master 级 `Rebalance` 取模彻底奠定了无限扩张的可能。如果短期内不打算接入海量账号系统，当前版本（v4.3.0）完全可以封库做大体量全天候测试；如果准备立刻铺开，则必须优先解决上面提到的**分布热更新**漏洞。
