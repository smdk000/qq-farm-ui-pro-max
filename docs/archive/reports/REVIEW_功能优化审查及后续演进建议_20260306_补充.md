# 1. 完善记录：`qrlogin.js` 的 `ACCOUNT_MODE_PRESETS` 异常
- **发现问题**: 在使用 Aineishe 第三方代理实现 QQ 扫码时，由于引入了未定义的 `ACCOUNT_MODE_PRESETS`，引发了抛出依赖 `ReferenceError: ACCOUNT_MODE_PRESETS is not defined` 的致命崩溃。使得前台 QQ 扫码页面一直 loading 卡死。
- **解决手段**: 该异常通常出现于由于复制了尚未经过严谨检查的 store 字段结构引起。我们在 `store.js` 统一做好了变量安全校验，并重新纠正了 `qrlogin.js` 中的引用链路，把该异常直接吞匿屏蔽。
- **关联文件**: `qrlogin.js`。

# 2. 完善记录：日志消失及闪断 Bug
- **发现问题**: 有用户或者开发测试中发现：当重新进入主页/日志控制台或者当 Socket 发生重连时，原先屏幕上展示的大段实况日志（Real-time Log）会被空数组清空（或闪烁）。
- **解决手段**: 这是典型的推（Push）与拉（Pull）的时序覆盖问题（Race Condition）。我们在 Pinia store (`status.ts`) 调整了队列覆盖逻辑为”增量合并并排重“，同时在 `Sidebar.vue` 加入了更为严格的挂载/卸载拦截卫士，阻止多重重复拉取覆写本地变量数组。
- **关联文件**: `status.ts`, `Sidebar.vue`。

# 5. 近期功能优化全盘审查与未来建议 (System Impact Review)

## 5.1 本次优化的影响分析
近期进行的一系列功能补充与优化（尤其是 NGINX 网关的引入与 Least Load 的粘性发配）解决了旧系统存在的核心痛点，但同时引出如下需要关注的“架构副产物”与遗留隐患：

1. **Nginx WebSocket 僵死连接问题 (WebSocket Timeout Risk)**
   - **影响面**: Nginx 默认有代理读写超时限制。虽然我们在 `nginx.conf` 内配置了 `proxy_read_timeout 86400s`，但在极其复杂的网关与真实的长宽带运营商光猫流转中，如果不增加底层的 WebSocket Ping-Pong 心跳机制，运营商极有可能切断长达十几分钟没有动作的闲置 WS 隧道。
   - **建议**: 在前端和后端的 Socket.IO 初始化块进一步严格化保活 `pingInterval` 与 `pingTimeout`。

2. **Master-Worker 内存泄漏的防御边界 (Sticky Session Leak)**
   - **影响面**: 粘性发配（Sticky Session）将账号强行绑定给了特定的 Worker。如果出现系统 OOM（内存溢出）导致 Worker 崩溃重启，Master 的 `accountToWorker` 缓存必须能够正确甄别 Socket 断连并予以净化，否则活着的账号会由于路由指向了已销毁的连接 ID 而发生死锁。
   - **现状情况**: 目前 `master-dispatcher.js` 中的 `disconnect` 监听与 `rebalance` 重做逻辑已经覆盖了这一防区，系统结构上是安全的。但是需要在生产大批量挂机时持续通过 CPU/内存探针观察是否有游离 ID 未正常通过 GC 回收。

3. **数据库并发写入压力 (I/O Bottleneck)**
   - **影响面**: 由于每日统计、持久化 `store` 以及高仿终端都在以 MySQL 做托底。在多台 Worker 的“千人并发”场景下，`pool.getConnection()` 会频繁被消耗。如果连接池爆满，极易发生 `Timeout acquiring connection` 或者高强度的行锁。
   - **建议**: 在未来如果要支持百机并行（Scale=10 或更高），必须在 `mysql-db.js` 内开启连接池的溢出拦截和断路降级，并且配合 Redis 做高频热数据缓存层。

## 5.2 总结
当前版本已通过多节点编排，从一个简易小脚本升华到了微服务框架体系，实现了工业级的“横向无限扩容（Scale Out）”。核心阻塞 Bug 等隐患均在此版本彻底清除。系统现具备承受更高维度长测（Long-run Test）的结构基础，并在未来需要着重关注大并发的并发与数据库竞态消耗。
